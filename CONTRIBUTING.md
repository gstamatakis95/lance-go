# Contributing to lance-go

Thanks for your interest in contributing. This document covers what you need
to build and test the project, the hard rules the codebase enforces, and the
checklist to run before opening a pull request. For the full architectural
tour (repo map, module-by-module breakdown, FFI conventions in depth), see
[AGENTS.md](AGENTS.md).

## Prerequisites

- Go 1.25+
- Rust 1.94+ (CI pins `1.94.0`)
- `protoc` on `PATH` (the `lance` crate generates protobuf code at build time)

## Building and testing

```sh
make rust               # build the Rust FFI shim (regenerates include/lance_go.h)
                        # cold build: ~6-25 min; shim-only rebuilds: seconds
make go-test            # go test ./... -race -count=1
make go-test-cgocheck   # GOEXPERIMENT=cgocheck2 go test ./... -count=1
make rust-test          # Rust workspace unit tests, release profile
make test               # rust build + Rust tests + both Go test runs
make lint               # cargo fmt --check, cargo clippy -D warnings, go vet
make fmt                # format Rust and Go sources
make header-check       # verify the checked-in header matches cbindgen output
make docs               # regenerate docs/api.md from package godoc
make docs-check         # verify docs/api.md is current (CI gate)
```

Object-store tests (S3/Azure/GCS emulators) are gated off by default; see
[AGENTS.md](AGENTS.md#build--test) if you need to touch that code path.

## Hard rules

These are enforced by CI or by the nature of the FFI boundary; violating them
will fail review or fail CI outright.

1. **Never hand-edit `include/lance_go.h`.** It is cbindgen output,
   regenerated by `rust/build.rs` on every `make rust`. Change the Rust
   source, rebuild, and commit the regenerated header together with the
   Rust change. CI fails on drift.
2. **Never run a bare `cargo update`.** `rust/Cargo.lock` pins `time = 0.3.47`
   to work around a rustc-1.94 / aws-smithy trait conflict. Update individual
   crates with `cargo update -p <crate>` and keep the `time` pin intact.
3. **All Arrow data exported to the native side must be C-allocated.** Use
   `lance.Allocator()` in application code; in tests, always use
   `internal/testutil.Allocator()`, never `memory.DefaultAllocator`.
   Violations crash under `GOEXPERIMENT=cgocheck2`, which CI runs.
4. **All fallible FFI calls must go through `ffiCall(ctx, ...)`.** Never call
   a fallible `C.lance_*` function directly; `ffiCall` pins the OS thread
   across the call and the thread-local error retrieval, and installs the
   ctx cancellation token.
5. **Go errors wrap exactly one sentinel** from `lance/errors.go`, with the
   `"lance: <verb>:"` prefix added exactly once at the wrap layer, so
   `errors.Is` classification keeps working and error strings never
   double-prefix.
6. **Docstring standard: every exported symbol is documented, and the
   comment starts with the symbol's name** (standard Go doc-comment
   convention), since `docs/api.md` is generated from these comments via
   `make docs`.

See [AGENTS.md](AGENTS.md#hard-rules) for the callback-bridge-specific rules
(payload ownership, panic containment, release semantics) if you're touching
`lance/callbacks.go` or `rust/src/callbacks.rs`.

## Commit conventions

- Imperative subject, <= 72 chars, no trailing period. Body explains *why*
  when the diff doesn't.
- Rust changes that alter the FFI surface must include the regenerated
  `include/lance_go.h` in the same commit.
- Never commit `rust/target/`, `dist/`, or downloaded artifacts
  (`lib/{os}_{arch}/`).

## Before opening a pull request

Run the same gates CI runs:

```sh
make lint
make test
make docs-check   # if you changed exported Go doc comments, run `make docs` first
```

All three should be green. If `make docs-check` fails, run `make docs` and
commit the regenerated `docs/api.md` alongside your change.
